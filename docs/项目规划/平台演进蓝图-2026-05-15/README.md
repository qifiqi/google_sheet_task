# 平台演进方案目录

更新时间：2026-05-15

本目录专门用于沉淀两类文档：

1. 基于当前仓库真实结构的任务平台演进方案
2. 基于当前仓库任务特点的多 AI agent 并行开发协作方案

这里的重点不是抽象的产品蓝图，而是围绕当前项目上下文来写：

- `run.py` + `app/startup.py` 的真实启动链路
- `app/services/task/facade.py` + `app/services/task/*` 的真实任务主控结构
- `scheduler_service.py` / `scheduled_task_worker.py` / `task_watchdog.py` 的现有执行与恢复能力
- C31 批量拆分、Google Sheet token / sheet 占用、长任务恢复这些仓库特有问题

目录说明：

- `01-基于当前平台上下文的架构演进方案.md`
  - 直接面向这次“一次切换式大改”，明确新内核、数据模型、切换策略和实施顺序
- `02-多AI-Agent并行开发协作方案.md`
  - 讨论如何把这次重构拆成多个 AI agent 并行执行，仿照一个完整开发团队协作

建议阅读顺序：

1. 先看 `01-基于当前平台上下文的架构演进方案.md`
2. 再看 `02-多AI-Agent并行开发协作方案.md`

核心原则：

1. 不脱离现有任务链路、启动钩子、资源占用和恢复机制来谈架构
2. 接受新建 `app/platform/` 内核，并按一次切换方案推进
3. 多 AI agent 方案优先服务“开发协作”和“任务拆分”，不是先做产品内自治 Agent
