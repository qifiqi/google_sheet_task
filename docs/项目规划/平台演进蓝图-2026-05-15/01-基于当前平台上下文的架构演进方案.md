# 基于当前平台上下文的架构演进方案

更新时间：2026-05-15

---

## 1. 文档定位

这份文档不再走温和演进口径。  
它基于当前仓库上下文，直接给出一套“新建 `app/platform/` 内核、一次切换上线、旧入口保留只读”的激进重构方案。

当前已确认的决策前提：

- 当前暂时没有必须持续运行的线上长任务
- 接受数据库升级
- 接受前后端一起重做
- 接受一次切换，不做长期双轨
- 新内核放在 `app/platform/`
- 旧入口保留只读，不再创建新任务
- 主任务执行必须移出 Flask Web 进程
- 数据库为 `PostgreSQL`
- 当前以 Windows 为主，但未来要迁到 Linux / Docker
- 上传脚本执行采用“同版本包可复用 venv + 哈希缓存”
- checkpoint 第一版只强制支持 `Google Sheet` / `backtest`
- 接口按前后端分离方式设计，但前端技术栈暂不锁死

---

## 2. 总体结论

这次最优路线不是继续重构旧 `TaskManager`，而是：

1. 保留 `tasks` 作为任务定义主表
2. 新增 `task_executions` 作为执行主表
3. 新建 `app/platform/` 内核
4. 新前端、新 API 直接对接新内核
5. 旧任务数据保留只读
6. 旧执行链路冻结，不再承接新任务

一句话总结：

保留业务语义，替换执行内核。

---

## 3. 旧平台里要保留的“业务语义”，不是“代码结构”

这次大改不能简单理解为“重写任务系统”，因为旧平台里有几类语义是必须保留下来的。

### 3.1 必保留能力

- Google Sheet token 占用控制
- Google Sheet 资源占用控制
- C31 批量拆分能力
- backtest 任务能力
- 管理后台可查、可取消、可重启
- Google Sheet / backtest 的 checkpoint 继续执行能力

### 3.2 明确冻结的旧实现

以下旧实现不作为未来内核继续演进：

- `app/services/task/runtime.py` 的线程执行模型
- `Task.status` 单表承载定义态和执行态
- C31 仅前端特殊逻辑驱动的批量拆分方式
- `scheduler_service.py` 继续扩成新的业务执行入口
- `startup.py` 依赖大量补偿式恢复来保证一致性

## 4. 当前平台的真实结构

### 4.1 启动链路

当前平台的真实启动顺序是：

1. `run.py`
2. `app/__init__.py:create_app()`
3. `app/startup.py:bootstrap_app(app)`

`bootstrap_app(app)` 当前负责的事情很多，而且都是真正影响系统稳定性的动作：

- `db.create_all()`
- `ensure_google_sheet_token_schema()`
- `ensure_user_schema()`
- `ensure_task_schema()`
- `reset_google_sheet_token_occupancy()`
- `reset_google_sheet_occupancy()`
- `init_rbac()`
- `check_and_cleanup_dead_tasks()`
- `init_scheduler()`
- `init_task_watchdog()`

这意味着：

- 启动不是一个“轻壳”
- 平台当前依赖启动期修复来保证一致性
- 后续任何改造都不能绕开 `run.py` 和 `app/startup.py`

### 4.2 任务主控链路

当前任务真实入口已经不是旧的 `app/services/task_manager.py`，而是：

- `app/services/task/facade.py`
- `app/services/task/creation.py`
- `app/services/task/runtime.py`
- `app/services/task/restart.py`
- `app/services/task/occupancy.py`
- `app/services/task/query.py`

也就是说，平台已经有了一个“门面 + mixin”结构，只是执行模型仍旧是 Web 进程直接起线程。

### 4.3 业务任务类型

当前真正在线上结构里运转的任务至少包括：

- `google_sheet`
- `google_sheet_C4`
- `google_sheet_C5`
- `google_sheet_C31` 前端批量拆分后落成多个 `google_sheet` 子任务
- `backtest_training`

其中 C31 很特殊：

- 它不是一个独立执行器
- 它只是前端批量创建入口
- 最终在 `TaskManager.batch_create_and_start_task()` 中拆成多个 C3 子任务

### 4.4 调度与恢复链路

当前系统已经有两条很重要的“平台雏形”：

1. `app/services/scheduler_service.py`
2. `app/services/task_watchdog.py`

它们的现状分别是：

- `scheduler_service.py` 主要管理 `ScheduledTask`
- `scheduled_task_worker.py` 已经具备独立子进程执行的雏形
- `task_watchdog.py` 已经承担“发现异常运行态并重启”的职责

所以当前系统并不是没有调度和恢复，而是“主任务系统”和“定时任务系统”还没有统一成一套执行内核。

## 5. 新平台目标形态

新平台建议直接定成四层。

### 5.1 Web 控制层

职责：

- 页面
- API
- 权限
- 管理后台
- 任务定义编辑
- 执行历史查询

Flask 继续承担这一层。

### 5.2 Platform 内核层

职责：

- 任务定义
- 执行实例
- 批量编排
- 队列
- 调度
- 取消
- 重试
- 资源租约
- checkpoint 重启

这一层放在 `app/platform/`。

### 5.3 Worker 执行层

职责：

- 常驻 worker 主循环
- claim 任务
- 上报心跳
- 调执行器
- 释放租约

### 5.4 Executor 执行器层

第一版建议包含：

- `google_sheet`
- `backtest`
- `batch_orchestration`
- `python_script`
- `strategy_package`
- `data_cleaning`
- `reporting`

## 6. 新内核目录建议

建议直接新建：

```text
app/platform/
  domain/
  application/
  infrastructure/
  worker/
  executors/
  web/
  contracts/
```

### 6.1 `domain/`

放领域模型和核心规则：

- `task_definition.py`
- `task_execution.py`
- `resource_lease.py`
- `task_package.py`
- `restart_policy.py`

### 6.2 `application/`

放应用服务：

- 创建任务
- 创建执行实例
- 入队
- 批量扇出
- 取消
- 从头重跑
- checkpoint 重启

### 6.3 `infrastructure/`

放基础设施适配：

- PostgreSQL 仓储
- APScheduler 适配
- 日志落盘
- venv 管理
- 任务包缓存
- 文件存储

### 6.4 `worker/`

放 worker 运行循环：

- claim
- heartbeat
- shutdown/drain
- execution bootstrap
- failure handling

### 6.5 `executors/`

放执行器实现：

- `google_sheet/`
- `backtest/`
- `batch_orchestration/`
- `python_script/`
- `strategy_package/`
- `data_cleaning/`
- `reporting/`

### 6.6 `web/`

放新平台 API 适配层，供 Flask 蓝图使用。

### 6.7 `contracts/`

放枚举、DTO、任务包 manifest 规范、执行状态定义、重启模式。

## 7. 数据模型方案

### 7.1 保留 `tasks`

`tasks` 继续作为任务定义主表。  
它主要承载：

- 名称
- 描述
- 任务类型
- 配置
- 创建人
- 页面回填来源
- 模板来源

不再让 `tasks` 单独承担完整执行状态。

### 7.2 新增 `task_executions`

这是新平台的执行主表。

核心字段建议：

- `id`
- `task_id`
- `execution_no`
- `status`
- `queue_name`
- `priority`
- `trigger_type`
- `scheduled_at`
- `started_at`
- `finished_at`
- `worker_id`
- `heartbeat_at`
- `cancel_requested`
- `restart_mode`
- `current_step`
- `total_steps`
- `checkpoint_payload`
- `error_summary`
- `root_execution_id`
- `parent_execution_id`

### 7.3 新增 `task_execution_logs`

不要再让执行日志完全复用旧任务日志语义。  
建议按 execution 维度记录。

### 7.4 新增 `task_execution_artifacts`

用于保存：

- 导出结果
- 汇总文件
- 运行报告
- 上传脚本产物

### 7.5 新增 `resource_leases`

核心字段建议：

- `id`
- `resource_type`
- `resource_key`
- `owner_execution_id`
- `lease_status`
- `leased_at`
- `heartbeat_at`
- `expires_at`

用于统一管理：

- token 占用
- Google Sheet 占用

### 7.6 新增 `task_packages`

用于上传脚本和策略包。

核心字段建议：

- `id`
- `package_type`
- `package_hash`
- `manifest_json`
- `storage_path`
- `venv_cache_key`
- `created_by_user_id`

### 7.7 旧数据策略

历史旧任务：

- 保留
- 只读展示
- 不迁移旧执行态到新表

新任务：

- 全部进入新平台表体系

这样可以避免“历史执行态强迁移”带来的混乱。

## 8. PostgreSQL 下的并发与队列策略

数据库既然已经确定是 `PostgreSQL`，第一版就应该充分利用它的并发能力。

### 8.1 claim 策略

建议使用：

- `FOR UPDATE SKIP LOCKED`

适用于：

- 多 worker 并发领取任务
- 避免重复 claim
- 避免热点任务阻塞整个队列

### 8.2 幂等要求

每个 execution 都必须满足：

- 可唯一识别
- 可重复恢复
- 不因重复 claim 产生双执行

### 8.3 状态流转要求

必须做到：

- 每次执行状态变更都可审计
- `queued -> running -> completed/error/cancelled`
- `cancel_requested` 由数据库驱动，不依赖进程内事件对象

---

## 9. 调度、恢复与不中断策略

### 9.1 调度器定位

`APScheduler` 第一版保留。  
但它只做“触发器”，不直接承载复杂业务执行。

建议职责：

- cron 到点
- once 到点
- delayed 到点
- 创建 execution 并置为 `queued`

### 9.2 Worker 恢复策略

worker 异常退出后：

- 由 `heartbeat_at` 判定失联
- 由 lease 超时判定资源可回收
- 把 execution 标成可恢复或失败

### 9.3 减少中断的真实做法

你要求的是“减少中断”，不是零中断。  
因此推荐：

- worker 支持 drain 模式
- 发布新版本时先停领新任务
- 已运行任务尽量跑完
- 再切新 worker

这对 Windows 当前环境也更现实。

## 10. C31、Google Sheet、backtest 的新定位

### 10.1 C31

不再只是前端批量入口。  
新平台中建议升级成：

- 通用 `batch_orchestration` 执行器框架
- C31 只是该框架下的一个业务模板

这样以后：

- 批量回测
- 批量清洗
- 批量报表

都可以复用这一层。

### 10.2 Google Sheet

继续作为核心执行器之一。  
重点保留：

- token 占用控制
- sheet 占用控制
- checkpoint 恢复
- 网络可恢复错误分类

### 10.3 backtest

第一版按普通执行器纳入新平台。  
保留：

- checkpoint 继续
- 结果与日志查询
- 可重启

先不把它做成更复杂的 workflow 内核。

## 11. 上传脚本与策略包方案

### 11.1 支持两种输入形态

第一版统一支持：

- `single_script`
- `package_bundle`

其中：

- `single_script`：单 Python 脚本
- `package_bundle`：zip/目录，带 `manifest.json` + `entry.py`

### 11.2 依赖与 venv 策略

已确定策略：

- 同版本包可复用 venv
- 通过哈希缓存识别环境复用

建议缓存 key 由以下信息组成：

- Python 版本
- requirements 内容哈希
- manifest 中声明的 runtime 版本

### 11.3 Windows 到 Linux / Docker 迁移考虑

第一版设计时就要避免把执行环境写死在 Windows PowerShell 语义里。

建议：

- 平台层定义抽象 runner
- Windows 当前实现一套
- 未来 Linux / Docker runner 复用同一协议

## 12. 新前端与接口策略

前端技术栈暂不锁死，但接口必须按前后端分离方式设计。

这意味着：

- 新平台 API 必须独立于模板页面实现
- 管理后台与创建页都只依赖 API 契约
- 即便第一版仍用服务端模板，也不能把业务协议埋回内联 JS

### 12.1 管理后台分阶段范围

已确认先走 `C` 路线：

第一阶段先重做：

- 任务列表
- 任务详情
- 取消
- 重启
- 执行历史查看

第二阶段再补：

- 调度管理
- 脚本上传
- 资源占用监控
- 包管理

## 13. 一次切换实施方案

### 阶段 0：冻结旧平台新增功能

目标：

- 不再给旧执行链路加新任务类型
- 不再扩大旧线程执行模型影响面

### 阶段 1：先搭新内核骨架

交付：

- `app/platform/` 目录
- 新表结构
- execution 状态机
- worker 主循环
- PostgreSQL claim

### 阶段 2：接入核心执行器

交付：

- `google_sheet`
- `backtest`
- `batch_orchestration`

验收重点：

- token / sheet 占用闭环
- checkpoint 恢复闭环
- C31 逻辑可迁移到新平台

### 阶段 3：接入脚本与策略包

交付：

- `python_script`
- `strategy_package`
- venv 缓存
- 包管理

### 阶段 4：重做控制面

交付：

- 新 API
- 新管理后台核心页
- 新任务创建入口

### 阶段 5：切换与封存旧入口

切换动作：

- 新任务全部走新平台
- 旧入口改只读
- 旧执行链路不再承接新任务

---

## 14. 这次大改最重要的三条纪律

1. 不在旧 `app/services/task/*` 上继续做深层结构演进  
只能修 bug，不能继续把它当未来内核。

2. 先定新数据模型，再写 worker 和前端  
否则最后一定是表乱、状态乱、接口乱。

3. 旧数据只读保留，不强迁历史执行态  
否则切换风险会被历史包袱放大。

---

## 15. 最终结论

基于当前仓库上下文和你的决策，这次最合适的路线是：

- 保留 `tasks` 主表语义
- 新建 `task_executions` 等执行表
- 新建 `app/platform/` 内核
- 主任务全面 worker 化
- APScheduler 退回触发器角色
- C31 升级为通用批量编排框架
- backtest 第一版按普通执行器纳入
- 上传脚本/策略包采用 venv 复用 + 哈希缓存
- 新任务一次切换进新平台
- 旧入口只读保留

一句话总结：

这不是“改良旧 `TaskManager`”，而是“保留业务语义、重建执行内核”的一次切换式重构。
