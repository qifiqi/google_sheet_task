# 配置字段对齐清单

## 1. 目的

这份文档用于完成 B 方案阶段 0 的“页面与接口字段对齐”工作。

核心原则：

- 以 `C5 config` 作为目标结构
- 默认版 `google_sheet` 保留现有结构，但后续可以映射到统一 schema
- `C4` 只做兼容映射，不再作为独立长期结构维护

## 2. 当前任务类型

当前库中主要任务类型：

- `google_sheet`
- `google_sheet_C5`

说明：

- 本地数据库中已有大量 `google_sheet` 样本和数十条 `google_sheet_C5` 样本
- `google_sheet_C4` 当前使用频率极低，可按“过渡兼容”处理

## 3. 三套 config 结构现状

### 3.1 默认版 `google_sheet`

主要字段：

- `spreadsheet_id`
- `sheet_name`
- `token_type`
- `token_file`
- `token_json`
- `proxy_url`
- `parameters`
- `parameter_positions`
- `check_positions`
- `result_positions`

特点：

- 单表结构
- 偏传统 Google Sheet 参数写入模型
- 详情页编辑器围绕顶层 `spreadsheet_id/sheet_name` 展开

### 3.2 过渡版 `google_sheet_C4`

主要字段：

- `task_type`
- `token_type`
- `token_file`
- `token_json`
- `proxy_url`
- `count_mode`
- `market_type`
- `date_range_mode`
- `start_date`
- `end_date`
- `parameters`
- `sheets`

特点：

- 已经引入 `sheets[]`
- 页面和逻辑结构明显向 `C5` 靠拢
- 仍保留一些 C4 专属处理逻辑和配置项

### 3.3 主版本 `google_sheet_C5`

主要字段：

- `task_type`
- `token_type`
- `token_file`
- `token_json`
- `proxy_url`
- `count_mode`
- `price_mode`
- `market_type`
- `date_range_mode`
- `start_date`
- `end_date`
- `parameters`
- `sheets`

特点：

- 多表结构
- 有更完整的业务维度字段
- 应作为统一 schema 的基准

## 4. 字段对齐表

| 字段 | 默认版 | C4 | C5 | 统一结论 |
|---|---|---|---|---|
| `task_type` | 可无 | 有 | 有 | 统一保留 |
| `spreadsheet_id` | 顶层字段 | 旧结构兼容 | 旧结构兼容 | 统一迁移到 `sheets[0].spreadsheet_id` |
| `sheet_name` | 顶层字段 | 旧结构兼容 | 旧结构兼容 | 统一迁移到 `sheets[0].sheet_name` |
| `title` | 少量使用 | 可在 `sheets[]` 中存在 | 可在 `sheets[]` 中存在 | 统一放在 `sheets[].title` |
| `sheets` | 无 | 有 | 有 | 统一保留，作为主结构 |
| `token_type` | 有 | 有 | 有 | 统一保留 |
| `token_file` | 有 | 有 | 有 | 统一保留 |
| `token_json` | 有 | 有 | 有 | 统一保留 |

## 5. 前后端联调重点

### 5.1 详情页读取

`GET /api/tasks/<task_id>` 当前返回：

- `task.id`
- `task.name`
- `task.description`
- `task.status`
- `task.task_type`
- `task.config`

联调要求：

- 前端必须接受 `config` 既可能是对象，也可能是字符串的情况
- 前端进入编辑态前必须先做归一化

### 5.2 详情页保存

`PUT /api/tasks/<task_id>/config` 当前接收：

- `name`
- `description`
- `config`

联调要求：

- 前端提交时必须是完整配置，不要只提交局部字段
- 未编辑字段应保留
- `sheets` 至少保留一项

### 5.3 新建 / 重启回填

重启场景：

- 通过 `GET /api/tasks/<task_id>` 拉原任务配置
- 前端做结构归一化后再回填页面

联调要求：

- `C4`、`C5` 都要走统一归一化流程
- `C4` 只需要被映射成“接近 C5”的结构，不再扩展独立编辑模型

## 6. 统一收敛方向

### 6.1 统一目标结构

建议以 C5 为基准，形成统一结构：

```json
{
  "task_type": "google_sheet_C5",
  "token_type": "file",
  "token_file": "data/token.json",
  "token_json": "",
  "proxy_url": null,
  "count_mode": "n_plus_1",
  "price_mode": "kp_price",
  "market_type": "cn",
  "date_range_mode": ["full"],
  "start_date": null,
  "end_date": null,
  "parameters": [["code"], [1, 2], [3, 4]],
  "sheets": [
    {
      "spreadsheet_id": "xxx",
      "sheet_name": "Sheet1",
      "title": "Optional title"
    }
  ]
}
```

### 6.2 映射策略

默认版 `google_sheet`：

- 保持服务端运行逻辑不变
- 前端或中间层可在需要时映射为统一结构

C4：

- 如果存在顶层 `spreadsheet_id/sheet_name`，转成 `sheets[0]`
- 缺失的 `price_mode` 使用默认值补齐
- 统一看成“弱化版 C5”

C5：

- 直接作为目标结构

## 7. 阶段 0 结论

当前最合理的策略是：

1. 把 `C5` 作为统一 schema 基准
2. 把 `C4` 降级为兼容映射层
3. 后续文档、前端归一化函数、服务抽象都优先围绕 `C5`
4. 默认版暂不强拆，但逐步与统一配置契约接轨
