# 阶段1-代码落地记录

## 本次目标

将统一 `config schema` 从文档落到代码入口，先控制“任务创建”和“任务配置更新”两条写入链路，再开始拆分 `TaskManager` 的读侧职责。

## 已完成

### 1. 新增统一配置模块

新增文件：

- `app/services/config_schema.py`

当前已实现：

- `normalize_task_config(config, task_type)`
- `validate_task_config(config, task_type)`

当前策略：

- `google_sheet_C5` 作为统一目标结构
- `google_sheet_C4` 按 C5 思路做兼容映射
- 默认 `google_sheet` 暂时保留现状，仅做最小校验

### 2. 接入任务创建入口

涉及文件：

- `app/routes/api.py`
- `app/services/task_manager.py`

处理方式：

- API 创建任务前先归一化，再校验
- `TaskManager.create_task()` 再做一次兜底归一化和校验

结果：

- 前端即使传入半旧结构，也会先被收敛后再落库
- 后续新入口接入时，不容易绕过统一契约

### 3. 接入任务配置更新入口

涉及文件：

- `app/routes/api.py`
- `app/services/task_manager.py`

处理方式：

- 详情页编辑提交的数据在 API 层先归一化，再校验
- `TaskManager.update_task_config()` 再做一次兜底处理

结果：

- 详情页编辑不再轻易把旧结构直接写回数据库
- 配置会逐步向统一结构收敛

### 4. 抽离 TaskManager 查询侧

涉及文件：

- `app/services/task_query_service.py`
- `app/services/task_manager.py`

处理方式：

- 新增 `TaskQueryService`，集中承接以下只读能力：
  - `get_task_status`
  - `get_all_tasks`
  - `get_task_logs`
  - `get_task_results`
  - `check_local_task_status`
- `TaskManager` 对外方法名和返回结构保持不变，内部统一委托到 `task_query_service`
- 查询服务不直接依赖运行态对象，由 `TaskManager` 传入 `running_tasks`、`get_task_logs`、`_get_config`，先保证兼容

结果：

- `TaskManager` 开始从“大而全”收敛到“任务编排与生命周期控制”
- 查询类 API 后续可以复用同一层查询服务
- 为下一步继续拆写侧、状态恢复和仓储层留出边界

### 5. 清理未接入的 RESTX 路由文件

涉及文件：

- `app/routes/api_restx.py`

处理方式：

- 检查 `create_app()` 的蓝图注册流程
- 检查项目内是否存在 `api_restx.py` 的导入、注册或间接引用
- 确认该文件当前未接入运行链路后，直接移除，避免与现有 `app/routes/api.py` 形成重复维护

结果：

- 统一保留当前真实生效的 Blueprint API 实现
- 消除一份未接线但内容重复的历史接口文件
- 后续接口优化只需要围绕 `app/routes/api.py` 和对应服务层继续推进

### 6. 收口前端配置归一化逻辑

涉及文件：

- `templates/google_sheet/base.html`
- `templates/google_sheet_c5/create.html`
- `templates/google_sheet_c5/detail.html`

处理方式：

- 在基类模板中新增共享函数 `normalizeTaskConfigForC5()` 与 `parseTaskConfigObject()`
- 创建页改为直接复用共享归一化入口，不再单独维护一份 `normalizeC5Config`
- 详情页读取和编辑弹窗回填也改为使用同一套解析规则，保证创建页与详情页的配置兼容行为一致

结果：

- C5 页面层的旧结构兼容逻辑从“多处散落”收敛为“单点维护”
- 后续如果 schema 再增加字段，只需要优先维护共享解析函数
- 降低创建页和详情页出现字段默认值不一致的风险

### 7. 补齐模板链路的统一 schema

涉及文件：

- `app/routes/api.py`
- `templates/google_sheet_c5/create.html`

处理方式：

- 模板列表按 `task_type` 过滤时，先对模板配置做统一归一化，避免旧模板因历史字段结构不同被漏掉
- 模板创建、模板更新接口接入 `normalize_task_config()` 和 `validate_task_config()`
- 模板详情返回前先做一次归一化，前端回填时直接使用统一结构
- 修正创建页模板列表加载时的返回值判断逻辑，避免前端错误放过异常响应

结果：

- 模板保存、模板回填、任务创建三条链路现在共享同一套 schema
- C5 模板即使是历史旧结构，也更容易被正确识别和回填
- 降低模板筛选、模板加载、创建页回填之间的结构漂移风险

### 8. 开始拆分 TaskManager 写侧持久化职责

涉及文件：

- `app/services/task_repository.py`
- `app/services/task_manager.py`

处理方式：

- 新增 `TaskRepository`，集中承接任务创建、重启任务复制、任务配置写回、历史结果清理、级联删除等数据库持久化动作
- `TaskManager` 继续保留流程编排、状态控制、线程管理和业务约束校验
- 先抽离低风险的持久化方法，避免一次性把运行态逻辑和数据库逻辑同时搬动

结果：

- `TaskManager` 进一步从“既管流程又管落库”收敛到“以编排为主”
- 后续如果继续拆 `TaskStatusService` 或 `TaskRuntimeRegistry`，边界会更清晰
- 写侧重构已经进入 B 方案阶段 2，而不是停留在只拆读侧

### 9. 抽离运行态注册表

涉及文件：

- `app/services/task_runtime_registry.py`
- `app/services/task_manager.py`
- `app/routes/api.py`

处理方式：

- 新增 `TaskRuntimeRegistry`，集中管理任务线程和事件队列
- `TaskManager` 改为通过注册表创建、登记、清理线程与事件队列
- API 的 SSE 与确认事件入口也改为优先通过注册表读取运行态资源

结果：

- `TaskManager` 不再直接承担运行态资源容器的主要维护职责
- 线程注册、事件队列注册、SSE 读取这几类操作开始有独立边界
- 阶段 2 已从“拆读侧 + 拆写侧”继续推进到“拆运行态”

### 10. 开始抽离任务状态判断服务

涉及文件：

- `app/services/task_status_service.py`
- `app/services/task_manager.py`

处理方式：

- 新增 `TaskStatusService`，集中处理取消任务、重启任务、配置编辑前的状态校验
- 将重启原因生成、断点重启计划生成从 `TaskManager` 中抽离
- `TaskManager` 改为组合查询服务、仓储层、运行态注册表和状态服务，进一步收敛到编排职责

结果：

- 取消、重启、配置编辑这几类典型状态判断开始具备独立边界
- `TaskManager.restart_task()` 中的分支判断明显收缩，后续更适合继续补自动化测试
- 阶段 2 已继续向“状态流转逻辑拆分”推进
