# 登录与鉴权操作指南

本文用于快速说明项目当前的登录、鉴权与单点登录行为，覆盖前端与后端协作方式。

---

## 1. 鉴权开关与基础配置

在环境变量中设置：

```bash
AUTH_ENABLED=true
JWT_SECRET_KEY=请替换为生产随机密钥
```

- `AUTH_ENABLED=true`：启用 JWT + RBAC 鉴权
- `AUTH_ENABLED=false`：仅用于本地调试，接口会绕过登录与权限检查

---

## 2. 登录流程（前后端协作）

1. 前端登录页调用 `POST /api/auth/login`
2. 后端校验账号密码并签发：
   - `access_token`（2 小时）
   - `refresh_token`（7 天）
3. 前端将 token 保存到 `localStorage`
4. 后续请求通过 `Authorization: Bearer <access_token>` 调用受保护接口

---

## 3. 单点登录机制（关键）

项目已启用“同账号单会话”策略：

1. 每次登录成功，后端都会递增用户 `token_version`
2. 新签发的 access/refresh token 会写入当前 `token_version`
3. 接口访问与 refresh 时都会比对 token 里的版本与数据库版本
4. 如果版本不一致，直接返回 401

结果：
- 新设备登录后，旧设备会在下一次请求时失效并回到登录页

---

## 4. 前端操作规范

### 4.1 必须做法

- 统一使用 `frontend/src/api/index.js` 导出的 `api` 实例
- 不要在业务页面直接使用裸 `axios`
- 通过 `useAuth()` 管理登录态（`login / logout / refresh / fetchUser`）

### 4.2 401 处理

- 请求返回 401 时，拦截器自动尝试 `POST /api/auth/refresh`
- refresh 成功：重试原请求
- refresh 失败：清理 token，跳转 `/login`

---

## 5. 后端操作规范

### 5.1 接口鉴权装饰器

```python
@bp.route('/api/xxx', methods=['GET'])
@login_required
@permission_required('task:view')
def xxx():
    ...
```

- `@login_required`：校验 token、用户状态、`token_version`
- `@permission_required`：校验权限码（OR 逻辑）

### 5.2 推荐接口划分

- 公开接口：不加 `@permission_required`（按需决定是否登录）
- 业务接口：至少加 `@login_required`
- 管理接口：`@login_required + @permission_required`

---

## 6. 常用鉴权接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/login` | 登录 |
| POST | `/api/auth/refresh` | 刷新 access_token |
| GET | `/api/auth/me` | 获取当前用户 |
| POST | `/api/auth/logout` | 退出登录并失效当前会话 |
| PUT | `/api/auth/password` | 修改密码 |

---

## 7. 快速联调示例

```bash
# 登录
curl -X POST http://localhost:5000/api/auth/login \
  -H "Content-Type: application/json" \
  -d "{\"username\":\"admin\",\"password\":\"admin123\"}"
```

```bash
# 携带 access_token 访问接口
curl http://localhost:5000/api/auth/me \
  -H "Authorization: Bearer <access_token>"
```

## 8. 常见问题

- **为什么旧端突然 401？**  
  通常是同账号在其他端重新登录，触发单点登录失效。

- **为什么本地不用登录？**  
  检查是否设置了 `AUTH_ENABLED=false`。

- **刷新总是失败怎么办？**  
  检查 `JWT_SECRET_KEY` 是否变更、refresh_token 是否过期、或 token_version 是否已被新登录覆盖。
