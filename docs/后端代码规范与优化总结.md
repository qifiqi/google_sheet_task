# 后端代码规范与优化总结

## 1. 调度模块（SchedulerService）

- **位置**
  - 服務：`app/services/scheduler_service.py`
  - 接口：`app/routes/scheduler_api.py`

- **主要问题与风险**
  - `running_tasks` 使用普通 `dict` 记录异步任务状态，存在并发读写风险。
  - 路由层（`/api/admin/scheduler/tasks/<id>/run`）直接调用内部方法 `_execute_task_async`，封装边界不清晰，后续内部实现变更容易影响接口。

- **已做优化**
  - 新增 `self._tasks_lock = threading.Lock()`，对 `running_tasks` 的所有读写增加锁保护：
    - 在 `_execute_task_async` 中创建和更新任务状态时加锁。
    - 在 `get_async_task_status` 和 `cleanup_completed_tasks` 中遍历和删除任务状态时加锁，避免竞态。
  - 新增公开方法 `run_job_once(self, task_id)`：
    - 统一对外提供“立即异步执行一次任务”的入口。
    - 内部使用 `_execute_task_async`，并增加运行状态校验与错误日志。
  - `scheduler_api.py` 中 `/api/admin/scheduler/tasks/<int:task_id>/run` 接口改为调用 `scheduler_service.run_job_once(task_id)`：
    - 不再直接依赖内部方法 `_execute_task_async`。
    - 根据返回值判断是否提交成功，失败时返回 500 和友好提示。

- **影响接口**
  - `POST /api/admin/scheduler/tasks/<int:task_id>/run`
  - 查询调度与异步任务状态的接口：
    - `GET /api/admin/scheduler/tasks/<int:task_id>/status`
    - `GET /api/admin/scheduler/status`
    - `GET /admin/api/scheduler/status`（管理后台使用）

- **后续建议**
  - 若后续新增其他“立即执行一次”的需求，统一复用 `run_job_once`，保持接口层与内部实现解耦。
  - 如需要更复杂的状态上报，可以在 `running_tasks` 中增加结构化字段（如任务名、参数摘要等）。

## 2. 日志读取与日志接口

- **位置**
  - 工具：`app/utils/log_reader.py`（新增）
  - 接口：`app/routes/api_restx.py` 中 `logs_ns` 相关接口

- **原有问题**
  - 多处接口（旧 `api.py` 与 RESTX `api_restx.py`）中直接：
    - `open(Config.LOG_FILE).readlines()` 全量读取日志文件。
    - 使用重复的正则和解析逻辑解析日志行。
  - 问题包括：
    - 性能与内存开销较大（日志大时风险更明显）。
    - 解析逻辑重复，维护成本高，修改一处需多点同步。

- **统一日志读取工具 `log_reader.read_logs`**
  - 文件：`app/utils/log_reader.py`
  - 主要能力：
    - 从 `Config.LOG_FILE` 读取日志，支持：
      - `limit`：返回条数限制。
      - `level`：按日志级别过滤。
      - `search`：按关键字检索消息。
      - `date_prefix`：按日期前缀过滤（例如 `2025-12-31`）。
      - `task_id` + `task_only`：基于任务 ID 过滤相关日志，支持 `[Task-xxxx]` 前缀和完整任务 ID 匹配。
      - `since`：只返回指定时间之后的日志（用于增量拉取）。
    - 内部统一使用正则表达式解析行：
      - 正常解析到结构化字段：`timestamp`、`level`、`message`、`source`。
      - 无法解析的行以降级方式返回，保留原始文本，避免数据丢失。
    - 性能优化：只对尾部有限倍数的行进行解析（`limit * N`），避免处理整个大文件。

- **RESTX 日志接口改造**
  - 受影响的 namespace：`logs_ns` in `app/routes/api_restx.py`

  1. `GET /api/logs`
     - 原实现：手动读取文件 + 正则解析 + 一系列过滤逻辑。
     - 现实现：统一调用 `read_logs`：
       - 参数映射：
         - `limit` → `limit`
         - `level` → `level`
         - `search` → `search`
         - `date` → `date_prefix`
         - `task_id` → `task_id`，并设置 `task_only=True`（仅返回与任务相关的日志）。

  2. `GET /api/logs/latest`
     - 原实现：从尾部读取 `limit * 2` 行，基于 `since` 过滤并返回最新日志。
     - 现实现：直接调用 `read_logs(limit=limit, since=since)`，由工具完成解析和过滤。

  3. `GET /api/tasks/<task_id>/system-logs`
     - 原实现：整文件扫描 + 任务相关字符串匹配 + 解析。
     - 现实现：调用 `read_logs(limit=limit, level=level_filter, task_id=task_id, task_only=True)`，统一行为。

- **影响接口**
  - `GET /api/logs`（RESTX 日志列表）
  - `GET /api/logs/latest`（RESTX 最新日志）
  - `GET /api/tasks/<string:task_id>/system-logs`（RESTX 任务相关日志）

- **后续建议**
  - 旧版 Blueprint 接口 `routes/api.py` 中也有类似日志接口，如继续使用，可逐步迁移为调用 `log_reader.read_logs`，以减少重复实现。
  - 若后续有更大规模日志需求，可考虑：
    - 将日志导入专门的日志系统（如 ELK/ClickHouse），当前工具函数可以作为过渡方案。

## 3. 接口与返回结构的一致性（建议方向）

> 本部分目前主要是规范建议，尚未对所有接口强行调整，仅在调度和日志相关接口中尽量保持一致。

- **现状**
  - 旧接口（`routes/api.py`）大量使用 `jsonify`，返回结构多为：`{"status": "success|error", ...}`，并根据情况返回 200/400/500。
  - RESTX 接口（`api_restx.py`）大多直接返回 dict，由 Flask-RESTX 统一 JSON 化，字段名与旧接口类似，但部分错误码和字段略有差异。

- **建议约定（后续逐步推广）**
  - 成功：
    - HTTP 状态码 200。
    - 响应结构统一包含：`status: 'success'`、`message`（可选）、`data` 或具体业务字段（如 `task`、`results` 等）。
  - 业务错误（参数错误、状态不允许等）：
    - HTTP 状态码 400。
    - `status: 'error'`，`message` 中给出明确错误原因。
  - 资源不存在：
    - HTTP 状态码 404。
    - `status: 'error'`，`message: 'xxx 不存在'`。
  - 服务器异常：
    - HTTP 状态码 500。
    - `status: 'error'`，`message` 可为简短信息，详细错误记录在系统日志中。

## 4. 其他已识别但暂未重构的点（后续可选优化）

> 这些部分暂未在本次改动中直接修改，只是在代码审查中识别出的问题，后续可按需要迭代。

1. **旧任务接口与新 RESTX 接口重复**
   - 旧接口：`app/routes/api.py` 下的一系列 `/tasks`、`/templates`、`/results`、`/logs`、`/config` API。
   - 新接口：`app/routes/api_restx.py` 下的 RESTX namespace `/api`。
   - 建议：确认前端是否完全迁移到 RESTX；若是，可逐步废弃旧 Blueprint 接口，只保留 RESTX 版本，减少维护成本。

2. **GoogleSheetService 与通知耦合较重**
   - 文件：`app/services/google_sheet_service.py`。
   - 问题：业务 service 直接依赖 `app.notifier` 发送钉钉通知，导致：
     - 本地开发与单元测试不够方便。
     - 业务逻辑和通知逻辑耦合。
   - 建议：
     - 将通知逻辑抽象到独立的通知服务或工具函数，由上层（如 TaskManager 或路由层）根据执行结果决定是否通知。

3. **ConfigManager 的数据格式兼容逻辑**
   - 文件：`app/services/config_manager.py`。
   - 现状：`get_google_sheet_config` 内对 `parameter_positions` / `check_positions` / `result_positions` 做了多处 `dict → list` 兼容，代码略显分散。
   - 建议：
     - 提取为统一的 `_normalize_positions` 辅助函数，提升可读性和可维护性。

## 5. 本次改动小结

- **已直接修改的文件**
  - `app/services/scheduler_service.py`
    - 增加任务状态锁 `_tasks_lock`，封装公共方法 `run_job_once`，并改进 `running_tasks` 访问的并发安全性。
  - `app/routes/scheduler_api.py`
    - `run_scheduled_task_now` 接口改为调用 `run_job_once`，增强内部方法封装性与错误处理。
  - `app/utils/log_reader.py`（新增）
    - 抽象统一的日志读取工具，提供多种过滤与分页能力，避免多处重复文件解析逻辑。
  - `app/routes/api_restx.py`
    - `logs_ns` 下日志相关接口改造为使用 `read_logs`，实现逻辑收敛与性能优化。

- **可感知的行为变化**
  - 调度相关接口在高并发下的状态查询与清理更安全，不易出现 `KeyError` 或状态不一致的问题。
  - 日志接口对大日志文件更友好，解析逻辑集中管理，便于后续扩展和 bug 修复。

如需继续深入优化（例如完全统一所有返回结构、对 GoogleSheet 部分做更彻底的解耦），可以在此文档基础上追加新版章节，保持演进记录。
