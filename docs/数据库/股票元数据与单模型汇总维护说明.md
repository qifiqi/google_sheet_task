# 股票元数据与单模型汇总维护说明

本文记录“股票名”相关改动，重点覆盖 `stock_metadata` 表、单模型汇总索引中的 `stock_name` 字段、股票搜索接口持久化，以及批量补库脚本。后续如果单模型汇总页面或 CSV 导出中股票名为空，优先按本文链路排查。

## 功能目标

- 单模型汇总页面在 `产品/股票` 旁边展示 `股票名`。
- 单模型汇总 CSV 导出同步包含 `股票名` 列。
- 股票搜索接口查到的股票代码、股票名、市场信息会保存到数据库，供后续任务创建和索引重建复用。
- 历史任务如果没有保存 `stock_name`，可以通过批量补库脚本补齐 `stock_metadata`，然后重新构建单模型汇总索引。

## 数据表

### `stock_metadata`

新增股票元数据表，模型位于 `app/models.py` 的 `StockMetadata`。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `stock_code` | 股票代码或产品代码，入库时会转为大写 |
| `stock_name` | 股票名或产品名 |
| `market_type` | 业务市场类型，规范化为 `cn` 或 `us` |
| `exchange_market` | 东方财富返回的交易市场编号，例如 `0`、`1`、`105`、`106` |
| `security_type_name` | 证券类型名称 |
| `source` | 数据来源，例如 `codetable`、`suggest`、`task_config` |
| `raw_json` | 原始搜索结果 JSON |

唯一约束：

```text
stock_code + market_type
```

这意味着同一个代码在不同市场类型下可以各有一条记录，但正常使用中 A 股为 `cn`，美股及 ETF 等为 `us`。

### `task_result_summary_index.stock_name`

单模型汇总索引表新增 `stock_name` 字段，用于页面查询和 CSV 导出。

启动期兼容修复在 `app/startup.py`：

- `ensure_stock_metadata_schema()`
- `ensure_task_result_summary_index_schema()`

迁移文件：

- `migrations/versions/20260528_add_stock_metadata.py`

## 股票名来源顺序

重建单模型汇总索引时，股票名按以下顺序获取：

1. `TaskResult.parameters` 中的显式字段：
   - `stock_name`
   - `name_cn`
   - `product_name`
2. `Task.config` 中的显式字段：
   - `stock_name`
   - `name_cn`
   - `product_name`
3. 根据 `stock_code + market_type` 查询 `stock_metadata` 表。
4. 都找不到时，`stock_name` 为空。

注意：当前不会从 `Task.name` 中解析股票名。例如任务名是 `C5-600776-东方通信`，`东方通信` 不会被当成股票名来源。这是有意设计，避免任务名格式不稳定时把错误文本写入索引。

## 股票代码来源

不同任务类型提取股票代码时有不同优先级：

- C5 / `google_sheet_C5`：
  - 优先从 `TaskResult.parameters.stock_code` 获取。
  - 例如 `{"stock_code": "qqq", "year": 2022, "A1": 3.5, "B1": 3}` 会识别为 `QQQ`。
  - 缺失时才走已有的任务名代码解析逻辑。
- C3 / `google_sheet`：
  - 按原有 C3 参数和结果结构提取。
- C4 / `google_sheet_C4`：
  - 按 C4 结果结构提取。
- 回测 / `backtest_training`：
  - 按回测结果参数提取。

这里的“任务名解析”只用于股票代码兜底，不用于股票名。

## 市场类型规则

股票元数据服务位于 `app/services/stock_metadata_service.py`。

`market_type` 规范化规则：

- `cn`、`a`、`a股`、`ashare`、`china` => `cn`
- `us`、`en`、`美股`、`usa` => `us`
- 如果无法识别：
  - 纯数字股票代码 => `cn`
  - 非纯数字股票代码 => `us`

这可以兼容东方财富搜索结果中 `market` 经常是交易所编号的情况，例如 `105`、`106`、`0`、`1` 不会被直接当成业务市场类型。

## 搜索接口持久化

以下搜索接口会把返回结果同步写入 `stock_metadata`：

- `app/routes/stock_api.py`
  - `/api/search-stocks`
- `app/routes/backtest_training.py`
  - `/backtest-training/api/search-stocks`
- `app/routes/backtest_multi_product.py`
  - `/backtest-multi-product/api/search-stocks`

写入字段来自东方财富搜索结果：

- `code` / `stock_code`
- `shortName` / `name`
- `market` / `marketType`
- `securityTypeName`
- 原始搜索 payload

后续创建任务时，如果 `Task.config` 里只有 `stock_code` 没有 `stock_name`，`TaskCreationMixin` 会尝试从 `stock_metadata` 回填 `stock_name`、`exchange_market`、`security_type_name`。

## 单模型汇总页面与导出

主要实现文件：

- `app/services/model_summary_service.py`
- `templates/admin/model_summary.html`

页面变化：

- 表格在 `产品/股票` 旁边新增 `股票名` 列。
- 查询输入支持按 `stock_code`、`stock_name`、`task_name` 模糊匹配。
- CSV 导出新增 `股票名` 列。

如果页面上股票名为空，通常不是页面问题，而是以下原因之一：

- 历史 `TaskResult.parameters` 没有保存 `stock_name`。
- 历史 `Task.config` 没有保存 `stock_name`。
- `stock_metadata` 表中没有对应 `stock_code + market_type` 的记录。
- 已补齐 `stock_metadata`，但还没有重新构建单模型汇总索引。

## 批量补库脚本

脚本文件：

```text
scripts/bulk_import_stock_metadata.py
```

用途：

- 批量把股票元数据写入 `stock_metadata`。
- 支持只提供 `stock_code`，脚本会调用东方财富搜索接口补齐股票名和市场信息。
- 支持 CSV / JSON / JSONL / TXT。
- 支持带表名前缀的列名，例如 `task_result_summary_index.stock_code`。

### 只提供股票代码

准备一个文本文件，例如 `stock_codes.txt`：

```text
QQQ
AAPL
600519
002916
```

执行：

```powershell
python scripts/bulk_import_stock_metadata.py D:\path\stock_codes.txt
```

如果你已经在仓库根目录下运行，也可以直接写相对路径：

```powershell
python scripts/bulk_import_stock_metadata.py data\stock_codes.txt
```

脚本会：

1. 读取每一行 `stock_code`。
2. 自动去重。
3. 调用东方财富搜索接口。
4. 选择代码匹配的搜索结果。
5. 写入 `stock_metadata`。

如果某些代码查不到，例如特殊指数代码，脚本不会强行写入，需要手工准备 `stock_name` 后再导入。

### CSV 输入

CSV 可以包含完整字段：

```csv
task_result_summary_index.stock_code,task_result_summary_index.stock_name,task_result_summary_index.market_type,task_result_summary_index.exchange_market
600519,贵州茅台,cn,1
AAPL,Apple Inc.,us,105
```

执行：

```powershell
python scripts/bulk_import_stock_metadata.py D:\path\stocks.csv
```

带表名前缀的列会自动剥离：

```text
task_result_summary_index.stock_code -> stock_code
task_result_summary_index.stock_name -> stock_name
```

### JSON 输入

支持对象数组：

```json
[
  {
    "task_result_summary_index.stock_code": "AAPL",
    "task_result_summary_index.stock_name": "Apple Inc.",
    "task_result_summary_index.market_type": "us"
  }
]
```

也支持带 `rows` 或 `data` 的对象：

```json
{
  "rows": [
    {
      "stock_code": "600519",
      "stock_name": "贵州茅台",
      "market_type": "cn"
    }
  ]
}
```

### 预览解析

如果只想确认文件能读取，不写数据库：

```powershell
python scripts/bulk_import_stock_metadata.py D:\path\stock_codes.txt --dry-run
```

注意：`--dry-run` 只解析输入行数，不会调用搜索接口，也不会写库。

## 历史数据修复流程

历史任务补股票名推荐流程：

1. 准备历史股票代码列表。
2. 执行 `scripts/bulk_import_stock_metadata.py` 补齐 `stock_metadata`。
3. 进入单模型汇总页面，执行“重建索引”。
4. 检查页面和 CSV 导出中的 `股票名` 列。

如果重建后仍为空，优先检查：

- `stock_metadata` 是否有对应 `stock_code`。
- 对应记录的 `market_type` 是否和代码规则一致。
- `task_result_summary_index` 是否已经重新生成。

## 路径规则

脚本对输入文件路径的处理方式如下：

- 绝对路径：直接使用，例如 `D:\Users\Administrator\Desktop\谷歌参数批量校验\data\stock_codes.txt`
- 相对路径：相对于你执行命令时所在的当前目录，例如在仓库根目录执行时，`data\stock_codes.txt` 会被解析成仓库下的数据文件
- 只有文件名：也会按当前目录解析
- `~` 开头：会展开成当前用户目录

脚本内部会先做 `expanduser()`，再做 `resolve()`，所以最终都会变成一个绝对路径来读取。真正区分“原来地址”和“相对地址”的关键，不是脚本本身，而是你运行命令时所在的当前工作目录。
