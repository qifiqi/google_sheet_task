# 多品数据回测比例算法审查

本文记录多品数据回测中“产品比例”参与计算的当前实现、算法差异、风险指标和建议修正方向。

## 结论

当前多品数据回测的全局预览采用的是“指标层加权”：

```text
比例计算值 = 产品A指标 * 产品A比例 + 产品B指标 * 产品B比例 + ...
```

这不等价于真实组合回测的“收益序列层合成”。后续若要改成真实组合口径，应保留为通用 `return` 计算思路：

```text
组合每日 return = 产品A每日 return * 产品A比例 + 产品B每日 return * 产品B比例 + ...

再基于组合后的每日 return 结果重新计算全部指标。
```

因此，当前结果适合做多产品指标的加权展示，但不能严格代表一个按比例配置后的真实组合产品回测结果。

## 当前实现

### 执行阶段

多品任务会按“参数方案 -> 产品”逐个执行单品回测。

每个产品独立：

1. 使用自己的 Google Sheet。
2. 使用自己的股票代码和 K 线。
3. 读取 Sheet 产出的 `index_return` 和 `start_return`。
4. 调用单品回测指标计算逻辑生成 `calculate_metrics`。
5. 每个产品保存一条 `TaskResult`。

关键链路：

- `app/services/backtest_multi_product_service.py`
  - `BacktestMultiProductService._execute_products()`
  - 每一步保存产品级 `TaskResult`
- `app/services/backtest_training_service.py`
  - `_execute_parameter_combination()`
  - 从 Sheet 输出列读取 `index_return` 和 `start_return`
  - 调用 `self.xpl.get_calculate_metrics_v1(_return_date)`
- `app/services/xpl_service.py`
  - `_calculate_metrics_v1()`
  - 基于单产品收益序列计算年化、月收益、波动率、夏普、回撤、所提诺、卡玛等指标

### 全局预览阶段

全局预览读取每个产品已经算好的指标，再按比例加权。

关键代码：

- `app/services/backtest_multi_product_service.py`
  - `_weighted_display(values, ratios, value_type)`
  - 计算公式为 `sum(value * ratio / 100)`
- `templates/backtest_multi_product/global_preview.html`
  - `weightedPreviewValue(values, ratios, valueType)`
  - 页面修改比例后，本地预览也使用同样的指标层加权

当前算法可以简化为：

```text
产品A指标 = metric(A.return)
产品B指标 = metric(B.return)

展示结果 = 产品A指标 * A比例 + 产品B指标 * B比例
```

## 真实组合算法

如果业务含义是“多个产品按比例组成一个真实组合”，更合理的算法应该是先合成同日期收益结果，再统一计算指标。

当前文档只保留通用 `return` 口径：

```text
组合 return[日期] =
    产品A return[日期] * 产品A比例
  + 产品B return[日期] * 产品B比例
  + ...
```

这里的 `return` 表示后续业务最终选定的产品收益字段。当前系统底层可取到 `start_return`、`index_return`，但组合口径不强制同时生成两条组合序列；如果后续需要分别计算模型收益和指数收益，再把该通用公式映射到对应字段即可。

### 数据对齐

同一个参数方案下，需要按日期对齐所有产品的收益数据：

```text
date
product_index
return
ratio
```

如果日期不完全一致，需要明确策略：

- 严格模式：任一产品缺少日期则报错或剔除整天。
- 交集模式：只保留所有产品都有数据的日期。
- 补齐模式：缺失产品按 0 收益或最近值处理。

回测指标通常建议使用交集模式或严格模式，避免组合收益被隐式补齐污染。

### 组合收益

对每个日期计算：

```text
portfolio_return[date] =
    sum(product_return[date] * product_ratio / 100)
```

然后构造：

```json
[
  {
    "date": "2024-01-02",
    "return": 0.0015
  }
]
```

再调用对应的指标计算逻辑。若复用当前 `xpl_analyzer.get_calculate_metrics_v1()`，需要在调用前把通用 `return` 映射到该函数要求的字段结构；如果未来新增组合专用计算器，则可直接消费 `date + return`。

```python
portfolio_metrics = calculate_portfolio_metrics(portfolio_return_date)
```

真实组合算法可以简化为：

```text
组合收益结果 = A收益结果 * A比例 + B收益结果 * B比例

组合指标 = metric(组合收益结果)
```

## 两种算法为什么不同

只有线性运算满足：

```text
metric(A * wA + B * wB) = metric(A) * wA + metric(B) * wB
```

多数回测指标不是线性的，所以不能直接加权。

### 标准差和波动率

当前算法：

```text
组合波动率展示值 = A波动率 * A比例 + B波动率 * B比例
```

真实组合：

```text
组合波动率 = std(A收益结果 * A比例 + B收益结果 * B比例)
```

两者通常不相等。

两产品组合的波动率还取决于相关性：

```text
σ组合 = sqrt(wA^2 * σA^2 + wB^2 * σB^2 + 2 * wA * wB * ρAB * σA * σB)
```

当前指标层加权无法体现相关性。如果两个产品走势相互抵消，真实组合波动率可能更低；如果高度同涨同跌，真实组合波动率可能接近加权结果。

### 夏普和所提诺

夏普依赖收益均值和标准差：

```text
Sharpe = 年化收益 / 年化波动率
```

当前算法：

```text
组合夏普展示值 = A夏普 * A比例 + B夏普 * B比例
```

真实组合：

```text
组合夏普 = 组合年化收益 / 组合年化波动率
```

由于分母的组合波动率不是简单加权，所以夏普不能直接加权。

所提诺同理，只是分母换成下行标准差，也不能直接按产品所提诺比例加权。

### 最大回撤和修复天数

最大回撤依赖净值路径：

```text
drawdown = (历史最高净值 - 当前净值) / 历史最高净值
```

当前算法：

```text
组合最大回撤展示值 = A最大回撤 * A比例 + B最大回撤 * B比例
```

真实组合：

```text
先生成组合净值曲线，再从组合净值曲线中计算最大回撤
```

如果产品 A 下跌时产品 B 上涨，组合回撤会被抵消；指标层加权看不到这种路径抵消。

修复天数也依赖净值曲线何时重新创新高，因此不能对各产品修复天数加权。

### 胜率类指标

胜率类指标通常判断某个周期是否大于 0：

```text
月盈利百分比 = 盈利月份数 / 总月份数
月超额收益胜率 = 月超额收益 > 0 的月份数 / 总月份数
```

当前算法：

```text
组合胜率展示值 = A胜率 * A比例 + B胜率 * B比例
```

真实组合：

```text
先算组合每月收益，再判断组合每月是否盈利
```

例如 A 某月亏损、B 某月盈利，组合后可能盈利也可能亏损，取决于两者幅度和比例。胜率不能从单产品胜率直接加权得到。

## 受影响指标

### 相对安全或近似合理的指标

这些指标是线性或近似线性的，指标层加权通常比较容易解释：

- 年化收益
- 平均月收益率
- 年化超额收益
- 平均月超额

但仍要注意：如果年化收益来自净值路径复利或分段计算，严格组合结果也可能和指标层加权存在差异。

### 明显不应直接加权的指标

这些指标应基于组合收益序列重新计算：

- 月收益率波动率
- 月超额波动率
- 夏普比率
- 超额夏普
- 所提诺比率
- 超额所提诺比率
- 卡玛比率
- 年最大回撤
- 年最大超额回撤
- 最大修复天数
- 超额最大修复天数
- 盈利年份百分比
- 月盈利百分比
- 跑赢年份百分比
- 月超额收益胜率
- 超额回撤胜率

## 可能影响的页面和接口

### 页面

- `/backtest-multi-product/global-preview/<task_id>`
  - `比例计算-指数`
  - `比例计算-结果`
  - 页面上修改比例后的即时预览

### 接口

- `GET /backtest-multi-product/api/global-preview/<task_id>`
  - 返回 `weighted_index_value`
  - 返回 `weighted_result_value`
- `PUT /backtest-multi-product/api/global-preview/<task_id>/ratios`
  - 保存比例后重新返回加权预览
- `GET /backtest-multi-product/api/global-preview/<task_id>/export`
  - Excel 导出的“比例计算-指数 / 比例计算-结果”

### 测试

- `tests/test_backtest_multi_product.py`
  - 当前测试覆盖的是指标层加权行为
  - 如果后续改为真实组合算法，需要新增组合收益序列级测试

## 业务风险

当前实现如果被用户理解为“真实组合产品回测”，可能产生以下偏差：

- 风险指标偏差：波动率、夏普、所提诺可能偏高或偏低。
- 回撤指标偏差：无法体现产品之间涨跌抵消后的组合净值路径。
- 胜率指标偏差：无法判断组合后的月份或年份是否真实盈利、跑赢。
- 参数方案排序偏差：如果用户按夏普、回撤、胜率选择参数方案，可能选到指标层加权好看但组合真实表现较弱的方案。
- 比例调仓误导：页面调整比例后的即时预览只改变指标权重，不会重新计算组合路径。

## 建议修正方向

### 方案一：保留当前算法，但明确命名

如果业务只需要“产品指标加权展示”，建议在文档、页面标题或导出表头中明确：

```text
比例计算 = 产品指标加权，不等同于真实组合回测指标
```

优点：

- 改动小。
- 现有接口和测试基本不变。

缺点：

- 仍不能提供真实组合产品指标。

### 方案二：新增真实组合指标

保留当前“指标加权”，同时新增一组“组合重算指标”。

全局预览可展示：

```text
指标加权-结果
组合重算-结果
```

优点：

- 兼容旧展示。
- 用户能比较两种口径。

缺点：

- 页面和导出列会更宽。
- 需要保存或重建每个产品的每日收益序列。

### 方案三：将比例计算改为真实组合算法

把当前 `weighted_index_value` / `weighted_result_value` 的含义改为组合收益序列重算结果。

优点：

- 业务语义最准确。
- 适合真实组合产品回测。

缺点：

- 需要调整数据保存结构。
- 需要处理日期对齐、缺失数据、比例变更后重算。
- 现有测试和前端预览逻辑需要同步更新。

## 实现真实组合算法需要补齐的数据

当前 `TaskResult.result` 中保存了 `calculate_metrics`，但全局预览主要读取指标摘要。要支持真实组合重算，建议确保每个产品结果中可取到每日收益序列：

```json
[
  {
    "date": "2024-01-02",
    "index_return": 0.0012,
    "start_return": 0.0015
  }
]
```

可选存储位置：

- 存在 `TaskResult.result` 内，作为每个产品结果的一部分。
- 存入 `TaskResultReturn`，通过 `task_id + step_index + product_index` 关联。
- 新增专门的多品收益序列表，便于查询和重算。

如果只依赖已经汇总好的 `calculate_metrics`，无法准确还原真实组合指标。

注意：后续组合口径只需要保留“产品 return * 比例后求和”的通用算法。当前表字段是否保存为 `start_return`、`index_return` 或新增通用 `return` 字段，需要在正式改造前按业务口径确认。

## 推荐判断

如果产品经理或用户说的是“按比例组合多个产品”，应优先使用方案二或方案三。

如果用户只是想看“各产品指标按权重折算后的汇总表”，当前算法可以保留，但必须在命名和说明中避免把它称为真实组合回测。

从回测专业口径看，风险、回撤、夏普、所提诺、胜率等指标应基于组合收益序列重新计算。
