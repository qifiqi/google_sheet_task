# 多品数据回测比例算法审查

本文记录多品数据回测中“产品比例”参与全局预览计算的当前实现。当前代码已经采用真实组合口径：先在每日收益序列层合成组合收益，再把组合收益序列投入指标算法。

## 结论

当前多品数据回测全局预览的比例计算为：

```text
组合 index_return[日期] =
    产品A index_return[日期] * 产品A比例 / 100
  + 产品B index_return[日期] * 产品B比例 / 100
  + ...

组合 start_return[日期] =
    产品A start_return[日期] * 产品A比例 / 100
  + 产品B start_return[日期] * 产品B比例 / 100
  + ...

组合收益序列 -> xpl_analyzer.get_calculate_metrics_v1() -> 组合指标
```

页面里的含义为：

- 产品列第三栏 `模型结果（70%）`：单个产品自己的收益序列按该比例缩放后重新计算出的指标。
- `比例计算-指数`：组合后的 `index_return` 序列重新计算出的指数指标。
- `比例计算-结果`：组合后的 `start_return` 序列重新计算出的模型结果指标。

旧实现曾经是“每个产品收益先按比例缩放、各自跑指标、最后同名指标相加”：

```text
历史实现 = metric(产品A return * 产品A比例) + metric(产品B return * 产品B比例) + ...
```

该历史实现已经不再作为全局预览主口径，因为多数回测指标不是线性的，指标层相加不等价于真实组合回测。

## 当前实现

### 执行阶段

多品任务仍然按“参数方案 -> 产品”逐个执行单品回测。每个产品独立：

1. 使用自己的 Google Sheet。
2. 使用自己的股票代码和 K 线。
3. 读取 Sheet 产出的 `index_return` 和 `start_return`。
4. 调用单品回测指标计算逻辑生成 `calculate_metrics`。
5. 保存一条产品级 `TaskResult`。
6. 将原始每日收益序列保存到 `TaskResultReturn.returns_json`，并通过 `TaskResult.return_series_id` 关联。

执行阶段不提前固化组合指标，也不把比例后的单品指标写成全局预览结果。

保存层会把单品按比例后的 `weighted_calculate_metrics` 一并写入 `TaskResult.result`，这样产品列第三栏可以直接复用已算好的结果；全局组合列仍然在预览阶段按组合收益序列动态重算。

关键链路：

- `app/services/backtest_multi_product_service.py`
  - `BacktestMultiProductService._execute_products()`
  - `BacktestMultiProductService._save_task_result()`
  - `_build_returns_json()`
- `app/services/backtest_training_service.py`
  - `_execute_parameter_combination()`
  - 从 Sheet 输出列读取 `index_return` 和 `start_return`
  - 调用 `self.xpl.get_calculate_metrics_v1(_return_date)`
- `app/services/xpl_service.py`
  - `_calculate_metrics_v1()`

### 全局预览阶段

全局预览进入时动态计算组合指标：

1. 按 `parameter_group_index` 收集同一个参数方案下的产品结果。
2. 从 `TaskResult.return_series_id -> TaskResultReturn.returns_json` 读取每个产品原始收益序列。
3. 若历史任务没有 `return_series_id`，尝试从旧 `TaskResult.result` 中的 `return_date` 恢复。
4. 按日期取所有产品都有数据的交集。
5. 用当前比例合成组合 `index_return` 和组合 `start_return`。
6. 调用 `xpl_analyzer.get_calculate_metrics_v1(portfolio_return_date)`。
7. 将组合指标映射为全局预览行里的 `weighted_index_value` / `weighted_result_value`。

关键代码：

- `app/services/backtest_multi_product_service.py`
  - `_get_return_date_for_task_result()`
  - `_build_portfolio_return_date()`
  - `_build_portfolio_metrics()`
  - `build_multi_product_global_preview_payload()`
- `templates/backtest_multi_product/global_preview.html`
  - 页面修改比例后调用后端 `/calculate-ratios` 试算
  - 前端不做本地计算

### 日期对齐

当前采用交集策略：只保留所有产品都有收益数据的日期。

```text
date
product_index
index_return
start_return
ratio
```

如果某个参数方案下无法得到共同日期，`比例计算-指数` / `比例计算-结果` 显示 `-`，不会把缺失产品收益默认为 0，也不会回退成旧的指标相加结果。

## 接口口径

以下接口都使用同一套组合收益重算逻辑：

- `GET /backtest-multi-product/api/global-preview/<task_id>`
  - 使用数据库中已保存的 `task.config.products[].ratio` 动态重算。
- `POST /backtest-multi-product/api/global-preview/<task_id>/calculate-ratios`
  - 使用请求中的临时比例动态重算，不写数据库。
- `PUT /backtest-multi-product/api/global-preview/<task_id>/ratios`
  - 保存比例到 `Task.config`，然后基于原始收益序列重新返回预览。
- `GET /backtest-multi-product/api/global-preview/<task_id>/export`
  - 导出 Excel 与 API 预览一致；如果 URL 传入临时比例，则用临时比例重算导出。

## 数据库与兼容

当前改造不新增表、不需要迁移。

继续使用：

- `TaskResult.result`：保存单品指标摘要和 Sheet 输出。
- `TaskResult.return_series_id`：关联原始收益序列。
- `TaskResultReturn.returns_json`：保存 `dates`、`index_returns`、`start_returns`。
- `Task.config.products[].ratio`：保存当前产品比例。

兼容策略：

- 新任务直接使用 `TaskResultReturn.returns_json`。
- 历史任务如果没有 `return_series_id`，但旧结果 JSON 中仍有 `return_date`，则可参与组合重算。
- 如果无法取到收益序列，组合列显示 `-`。
- 历史的 `weighted_calculate_metrics` 不再作为全局预览主数据源。

## 为什么不能指标相加

只有线性运算才满足：

```text
metric(A * wA + B * wB) = metric(A) * wA + metric(B) * wB
```

多数回测指标不是线性的。

### 波动率、夏普、所提诺

真实组合波动率依赖产品之间的相关性：

```text
σ组合 = sqrt(wA^2 * σA^2 + wB^2 * σB^2 + 2 * wA * wB * ρAB * σA * σB)
```

因此夏普、所提诺这类依赖波动率或下行波动率的指标必须基于组合收益序列重新计算。

### 回撤和修复天数

最大回撤和修复天数依赖净值路径。产品 A 下跌时产品 B 上涨，真实组合净值可能被抵消；指标层加权无法体现这种路径关系。

### 胜率类指标

盈利年份百分比、月盈利百分比、跑赢年份百分比、月超额收益胜率等指标需要先得到组合后的月份或年份收益，再判断是否盈利或跑赢，不能从单产品胜率直接加权得到。

  - 覆盖日期交集策略。
  - 覆盖没有共同日期或没有收益序列时显示 `-`。

## 工作约束

修改多品比例逻辑时应同时检查：

- 原始收益序列是否仍被保存到 `TaskResultReturn.returns_json`。
- 全局预览、临时比例计算、保存比例返回、导出 Excel 是否走同一套组合重算逻辑。
- 前端文案是否仍暗示“指标相加”。
- 历史结果缺失收益序列时是否显示 `-`，不要用旧指标相加结果伪装成组合指标。
