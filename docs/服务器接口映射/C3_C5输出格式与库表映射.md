# C3 C5 输出格式与库表映射

## 1. 文档说明

本文档用于说明：

- C3 输出格式
- C5 输出格式
- 当前系统保存数据库的格式
- 如果对接外部服务，建议如何设计表
- 是否可以放在同一张表里
- 如果不能，建议如何拆表

---

## 2. 当前系统数据库保存方式

当前系统与任务结果相关的核心表如下：

### 2.1 `tasks`

用途：

- 保存任务主记录

核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string(36) | 任务 ID |
| `name` | string | 任务名称 |
| `description` | text | 任务描述 |
| `status` | string | 任务状态 |
| `task_type` | string | 任务类型 |
| `config` | text | 任务配置 JSON |
| `start_time` | datetime | 开始时间 |
| `end_time` | datetime | 结束时间 |
| `current_step` | int | 当前步骤 |
| `total_steps` | int | 总步骤 |
| `error_message` | text | 错误信息 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 2.2 `task_results`

用途：

- 保存每一步执行结果

核心字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 结果主键 |
| `task_id` | string(36) | 关联任务 ID |
| `step_index` | int | 步骤序号 |
| `parameters` | text | 输入参数 JSON |
| `result` | text | 输出结果 JSON |
| `return_series_id` | int | 关联收益曲线 `task_results_return.id` |
| `success` | bool | 是否成功 |
| `error_message` | text | 错误信息 |
| `timestamp` | datetime | 结果时间 |

### 2.3 `task_results_return`

用途：

- 设计上用于保存收益率时间序列

字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `task_id` | string(36) | 任务 ID |
| `stock_date` | string | 日期 |
| `index_return` | float | 指数收益 |
| `start_return` | float | 起始收益 |
| `returns_json` | text | 收益曲线 JSON，按列存储 `dates/index_returns/start_returns` |

说明：

- 新格式优先使用 `returns_json` 一行保存整段收益曲线，减少 `task_id`、日期等字段重复。
- `task_results.return_series_id` 指向对应的收益曲线行，结果页和全局预览优先从该 ID 读取。
- 旧的 `stock_date/index_return/start_return` 明细列保留兼容历史数据。
- 如果不需要按天 SQL 查询收益，推荐写入 `returns_json`：

```json
{
  "dates": ["2024-01-01", "2024-01-02"],
  "index_returns": [0.001, -0.003],
  "start_returns": [0.002, 0.001]
}
```

## 3. C3 当前输出格式

## 3.1 数据库存储格式

C3 每一步执行成功后，会向 `task_results` 写入一条记录。

保存逻辑：

- `parameters`：保存当前参数组合
- `result`：保存 Google Sheet 结果单元格集合

示例：

```json
{
  "id": 1,
  "task_id": "task-uuid",
  "step_index": 0,
  "parameters": [1, 10, 30],
  "result": {
    "B6": 1,
    "B7": 10,
    "B9": 30,
    "B10": 5,
    "B11": 8,
    "B12": 12,
    "I15": 0.12,
    "I16": 0.22,
    "I17": -0.08,
    "I18": 0.10,
    "I19": 0.18,
    "I20": -0.12,
    "I21": 123.45,
    "I22": 10.2,
    "I23": 0.09
  },
  "success": true,
  "error_message": null,
  "timestamp": "2026-04-21T09:10:00"
}
```

## 3.2 C3 实际导出字段

C3 还会额外组装一份结构化数据，调用外部接口：

- `InsertStockTemplateParam`

导出字段如下：

| 字段 | 中文说明 | 来源 |
| --- | --- | --- |
| `stock_no` | 股票代码 | `task.name` |
| `multiplier` | 倍数参数 | `result.B6` |
| `danbian` | 单边参数 | `result.B7` |
| `xiancang` | 限仓参数 | `result.B9` |
| `zhishu` | 指数参数 | `result.B10` |
| `smoothing` | 平滑参数 | `result.B11` |
| `bordering` | 边界参数 | `result.B12` |
| `multiplier_index` | 倍数索引 | 当前 step |
| `danbian_index` | 单边索引 | 当前固定为 0 |
| `xiancang_index` | 限仓索引 | 当前固定为 0 |
| `zhishu_index` | 指数索引 | 当前固定为 0 |
| `smoothing_index` | 平滑索引 | 当前固定为 0 |
| `bordering_index` | 边界索引 | 当前固定为 0 |
| `return_rate` | 收益率 | `result.I15` |
| `annualized_rate` | 年化收益率 | `result.I16` |
| `maxdd` | 最大回撤 | `result.I17` |
| `index_rate` | 指数收益率 | `result.I18` |
| `index_annualized_rate` | 指数年化收益率 | `result.I19` |
| `max_index_dd` | 指数最大回撤 | `result.I20` |
| `fee_total` | 总手续费 | `result.I21` |
| `fee_annualized` | 手续费年化 | `result.I22` |
| `year_rate` | 年度收益率 | `result.I23` |

## 4. C5 当前输出格式

## 4.1 数据库存储格式

C5 每一步执行成功后，会向 `task_results` 写入一条记录。

示例：

```json
{
  "id": 1,
  "task_id": "task-uuid",
  "step_index": 0,
  "parameters": {
    "stock_code": "600519",
    "A1": 5,
    "B1": 20,
    "year": "2025-2023",
    "Kline_key": "2025-2023",
    "kline": [
      {
        "stock_date": "2023-01-03",
        "stock_val": 10.2
      },
      {
        "stock_date": "2025-12-31",
        "stock_val": 12.8
      }
    ]
  },
  "result": {
    "sheet_id_1__C5主表": {
      "D2": 1.23,
      "E2": 4.56,
      "F2": 7.89,
      "index_return_xpl": 0.11,
      "start_return_xpl": 0.22
    }
  },
  "success": true,
  "error_message": null,
  "timestamp": "2026-04-21T09:20:00"
}
```

## 4.2 C5 输出特点

与 C3 相比，C5 有几个明显特点：

- `parameters` 是对象，不是简单数组
- `result` 是按 `sheet_id__title` 聚合的嵌套对象
- 结果中既有固定计算字段，也有动态单元格字段
- 更适合 JSON 存储，不适合完全平铺死字段

---

## 5. 是否可以放在同一张表

结论：

- 可以
- 但不建议只做一张完全平铺的明细表

最推荐的做法是：

- 一张统一主表
- 一张统一结果明细表
- 必要时再加类型扩展表

也就是：

1. `task_export_job`
2. `task_export_result`
3. `task_export_result_c3_ext`
4. `task_export_result_c5_ext`

---

## 6. 推荐方案一：统一主表 + 统一明细表

这是最推荐的方案。

## 6.1 主表：`task_export_job`

用于保存任务级信息。

```sql
CREATE TABLE task_export_job (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  task_name VARCHAR(255) NOT NULL,
  task_type VARCHAR(64) NOT NULL,
  status VARCHAR(32) NOT NULL,
  source_system VARCHAR(64) DEFAULT 'google-parameter-validator',
  created_at DATETIME NULL,
  start_time DATETIME NULL,
  end_time DATETIME NULL,
  current_step INT DEFAULT 0,
  total_steps INT DEFAULT 0,
  error_message TEXT NULL,
  raw_config JSON NULL,
  updated_at DATETIME NULL,
  UNIQUE KEY uk_task_id (task_id)
);
```

字段映射：

| 目标字段 | 来源 |
| --- | --- |
| `task_id` | `tasks.id` |
| `task_name` | `tasks.name` |
| `task_type` | `tasks.task_type` |
| `status` | `tasks.status` |
| `created_at` | `tasks.created_at` |
| `start_time` | `tasks.start_time` |
| `end_time` | `tasks.end_time` |
| `current_step` | `tasks.current_step` |
| `total_steps` | `tasks.total_steps` |
| `error_message` | `tasks.error_message` |
| `raw_config` | `tasks.config` |

## 6.2 明细表：`task_export_result`

用于保存每一步结果。

```sql
CREATE TABLE task_export_result (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  task_type VARCHAR(64) NOT NULL,
  step_index INT NOT NULL,
  success TINYINT(1) NOT NULL DEFAULT 1,
  error_message TEXT NULL,
  parameters_json JSON NULL,
  result_json JSON NULL,
  exported_at DATETIME NULL,
  created_at DATETIME NULL,
  UNIQUE KEY uk_task_step (task_id, step_index)
);
```

字段映射：

| 目标字段 | 来源 |
| --- | --- |
| `task_id` | `task_results.task_id` |
| `task_type` | 通过 `tasks.task_type` 关联得到 |
| `step_index` | `task_results.step_index` |
| `success` | `task_results.success` |
| `error_message` | `task_results.error_message` |
| `parameters_json` | `task_results.parameters` |
| `result_json` | `task_results.result` |
| `created_at` | `task_results.timestamp` |

适用场景：

- 对方服务先把数据完整收下
- 后续再慢慢解析 C3/C5 的个性字段

优点：

- 改造成本最低
- 对动态字段最友好
- 最不容易丢数据

## 7. 推荐方案二：统一表 + 类型扩展表

如果对方服务除了保存原始 JSON，还希望方便按字段查询，可以继续拆扩展表。

## 7.1 C3 扩展表：`task_export_result_c3_ext`

```sql
CREATE TABLE task_export_result_c3_ext (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  step_index INT NOT NULL,
  stock_no VARCHAR(64) NULL,
  multiplier DECIMAL(18,6) NULL,
  danbian DECIMAL(18,6) NULL,
  xiancang DECIMAL(18,6) NULL,
  zhishu DECIMAL(18,6) NULL,
  smoothing DECIMAL(18,6) NULL,
  bordering DECIMAL(18,6) NULL,
  return_rate DECIMAL(18,6) NULL,
  annualized_rate DECIMAL(18,6) NULL,
  maxdd DECIMAL(18,6) NULL,
  index_rate DECIMAL(18,6) NULL,
  index_annualized_rate DECIMAL(18,6) NULL,
  max_index_dd DECIMAL(18,6) NULL,
  fee_total DECIMAL(18,6) NULL,
  fee_annualized DECIMAL(18,6) NULL,
  year_rate DECIMAL(18,6) NULL,
  raw_result_json JSON NULL,
  UNIQUE KEY uk_task_step (task_id, step_index)
);
```

## 7.2 C5 扩展表：`task_export_result_c5_ext`

```sql
CREATE TABLE task_export_result_c5_ext (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  step_index INT NOT NULL,
  stock_code VARCHAR(64) NULL,
  param_a1 VARCHAR(64) NULL,
  param_b1 VARCHAR(64) NULL,
  year_label VARCHAR(64) NULL,
  kline_key VARCHAR(64) NULL,
  kline_start_date DATE NULL,
  kline_end_date DATE NULL,
  index_return_xpl DECIMAL(18,6) NULL,
  start_return_xpl DECIMAL(18,6) NULL,
  raw_parameters_json JSON NULL,
  raw_result_json JSON NULL,
  UNIQUE KEY uk_task_step (task_id, step_index)
);
```

适用场景：

- 对方服务要按 `stock_code`
- 按 `year_label`
- 按收益率字段检索、统计、排序

优点：

- 查询更方便
- BI 或报表更好做

缺点：

- 开发成本更高
- 动态字段仍然要保留 JSON

## 8. 如果只想做一张表，建议怎么做

如果对方服务当前就想简单点，只上一张结果表，也可以。

建议表：

```sql
CREATE TABLE task_export_result_all_in_one (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  task_type VARCHAR(64) NOT NULL,
  step_index INT NOT NULL,
  success TINYINT(1) NOT NULL DEFAULT 1,
  stock_code VARCHAR(64) NULL,
  param_a1 VARCHAR(64) NULL,
  param_b1 VARCHAR(64) NULL,
  year_label VARCHAR(64) NULL,
  index_return_xpl DECIMAL(18,6) NULL,
  start_return_xpl DECIMAL(18,6) NULL,
  raw_parameters_json JSON NULL,
  raw_result_json JSON NULL,
  created_at DATETIME NULL,
  UNIQUE KEY uk_task_step (task_id, step_index)
);
```

说明：

- 对 C3 来说，`stock_code` 可以放 `task.name`
- 对 C5 来说，`stock_code` 直接取 `parameters.stock_code`
- 其他放不下的动态字段，全部进 `raw_parameters_json`、`raw_result_json`

结论：

- 能做
- 但这更像“过渡方案”

### 来源

- 任务主表：`tasks`
- 结果表：`task_results`

### 建议映射

| 当前字段 | 建议目标字段 |
| --- | --- |
| `tasks.id` | `task_export_job.task_id` |
| `tasks.name` | `task_export_job.task_name` |
| `tasks.task_type` | `task_export_job.task_type` |
| `task_results.step_index` | `task_export_result.step_index` |
| `task_results.parameters` | `task_export_result.parameters_json` |
| `task_results.result` | `task_export_result.result_json` |
| `parameters.stock_code` | `task_export_result_c5_ext.stock_code` |
| `parameters.A1` | `task_export_result_c5_ext.param_a1` |
| `parameters.B1` | `task_export_result_c5_ext.param_b1` |
| `parameters.year` | `task_export_result_c5_ext.year_label` |
| `parameters.Kline_key` | `task_export_result_c5_ext.kline_key` |
| `parameters.kline[0].stock_date` | `task_export_result_c5_ext.kline_start_date` |
| `parameters.kline[1].stock_date` | `task_export_result_c5_ext.kline_end_date` |
| `result.*.index_return_xpl` | `task_export_result_c5_ext.index_return_xpl` |
| `result.*.start_return_xpl` | `task_export_result_c5_ext.start_return_xpl` |

说明：

- `result.*.index_return_xpl` 的 `*` 表示不同 sheet 键
- 如果一个 step 会同时产出多个 sheet 结果，扩展表可以：
  - 只存主结果
  - 或再拆一个 sheet 结果子表

## 10. 如果 C5 有多 sheet，建议再拆一张子表

这是更规范的设计。

### 子表：`task_export_result_c5_sheet`

```sql
CREATE TABLE task_export_result_c5_sheet (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64) NOT NULL,
  step_index INT NOT NULL,
  sheet_key VARCHAR(255) NOT NULL,
  spreadsheet_id VARCHAR(128) NULL,
  sheet_title VARCHAR(255) NULL,
  index_return_xpl DECIMAL(18,6) NULL,
  start_return_xpl DECIMAL(18,6) NULL,
  raw_sheet_result_json JSON NULL
);
```

适合场景：

- 一个 step 对多个 sheet 写结果
- 对方要按 sheet 维度查结果

---

## 11. 最终建议

如果你要给对方一个最稳妥的落地建议，我建议这样说：

1. 先做两张基础表：
   - `task_export_job`
   - `task_export_result`
2. 两张表先把原始 JSON 接住：
   - `raw_config`
   - `parameters_json`
   - `result_json`
3. 如果后面查询需求明确，再加扩展表：
   - `task_export_result_c3_ext`
   - `task_export_result_c5_ext`
   - 必要时 `task_export_result_c5_sheet`

这样做的好处是：

- 第一阶段落地最快
- 不会因为字段理解不完整而丢数据
- 后面要扩字段时也不用返工原始数据
