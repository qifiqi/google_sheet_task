# 任务管理平台权限架构重构方案（长期版）

## 1. 基于当前代码的诊断

### 1.1 当前权限模型
- 角色与权限是标准 RBAC：`User -> Role -> Permission`，权限字符串形如 `task:view`。见 `app/models.py`。
- 接口层大多使用 `@permission_required(...)` 做路由级鉴权。见 `app/utils/auth.py`、`app/routes/task_api.py`。
- 菜单与页面可见性通过前端 `template-auth.js` + `/api/meta/nav` 组合实现。见 `static/js/template-auth.js`、`app/routes/meta_api.py`。

### 1.2 当前核心矛盾
- 任务执行相关接口是共享接口 `/api/tasks/*`，但鉴权仍是“全局 task 权限”：
  - 例如：`/api/tasks/<task_id>/cancel` 只检查 `task:cancel`，未区分该任务是 `backtest_training` 还是 `google_sheet`。见 `app/routes/task_api.py`。
- 任务实体已经有 `task_type` 字段（`google_sheet`、`google_sheet_C4`、`google_sheet_C5`、`backtest_training`），但权限决策未用它做资源级约束。见 `app/models.py`、`app/services/task_manager.py`。
- 导致结果：给回测角色赋予任务操作权限后，可能误操作 Google 任务，无法满足“看不到/操作不了不该看的任务”。

## 2. 推荐的长期方案

## 2.1 方案结论
- 采用 **RBAC（角色授予） + ABAC（条件约束）**，并以策略方式统一执行（可称 PBAC 实现形态）。
- 核心思想：
  - RBAC 负责“可以做什么动作”；
  - ABAC 负责“在什么资源范围内可以做”（按 `task_type/version/owner` 等）。

## 2.2 为什么适合本项目
- 你的平台“任务类型持续增加、接口复用高”，纯 RBAC 会持续冲突。
- 纯 ABAC 复杂度高，不利于运维和角色配置。
- RBAC + ABAC 组合能同时兼顾可维护性和精细控制。

## 3. 权限语义设计（建议）

## 3.1 动作权限（RBAC）
- `task:view`
- `task:create`
- `task:cancel`
- `task:restart`
- `task:delete`

以上是“动作能力”，不再默认代表所有任务类型。

## 3.2 作用域条件（ABAC）
- `task_type`: `google_sheet` / `google_sheet_C4` / `google_sheet_C5` / `backtest_training`
- `version`: `c3/c4/c5`（可选，若 `task_type` 已足够可先不加）
- `owner_id`: 是否仅本人任务（按需启用）

## 3.3 页面与任务操作分离
- 页面访问权限继续独立：如 `google_sheet:c3`、`google_sheet:c4`、`google_sheet:c5`、`backtest:view`。
- 任务操作权限由“动作 + 作用域”决定，不与页面权限互相替代。

## 4. 数据模型重构建议

新增一张“角色权限规则”表（示意）：

- `role_permission_rule`
  - `id`
  - `role_id`
  - `permission_code`（如 `task:delete`）
  - `scope_json`（如 `{"task_type":["backtest_training"]}`）
  - `effect`（`allow`，预留 `deny`）
  - `priority`
  - `is_active`

说明：
- 先不动现有 `role_permissions`，保留兼容。
- 新表用于精细规则；老权限作为兜底。

## 5. 后端执行架构（必须统一）

## 5.1 增加统一授权服务
- 新增 `app/utils/authorization.py`（建议）：
  - `authorize(user, permission_code, context) -> bool`
  - `filter_tasks(user, tasks, action) -> tasks`

## 5.2 路由改造原则
- 所有 `/api/tasks/*` 进入业务前，必须构造 `context` 决策：
  - 从 `Task` 中取 `task_type`、`config.version`、`created_by`（如有）。
- 不再只依赖 `@permission_required('task:*')`。

## 5.3 列表接口必须做“结果过滤”
- `GET /api/tasks` 不仅判断“能否访问接口”，还要对结果按规则过滤。
- 否则会出现“能看见不该看的任务”。

## 6. 前端策略建议

## 6.1 菜单策略
- 菜单只消费后端 `/api/meta/nav` 的过滤结果。
- 父级无可见子项时隐藏（当前 `meta_api.py` 已具备这类处理，继续沿用）。

## 6.2 页面级 403 提示
- 页面拦截显示“所需权限 + 缺失权限”（当前已在改进中）。
- 前端提示仅用于体验，最终授权以后端为准。

## 7. 迁移路径（避免一次性爆炸）

### 阶段 A：并行兼容
- 保留旧 `role_permissions`。
- 新增规则表与授权服务，但先对 `/api/tasks` 做灰度接入。

### 阶段 B：全量切换
- `/api/tasks/*`、回测相关、Google 相关统一走新授权服务。
- 完成角色迁移脚本（旧权限 -> 新规则）。

### 阶段 C：收敛
- 移除路由里散落的“全局 task:* 直接判断”。
- 仅保留统一授权入口与少量快捷装饰器。

## 8. 角色配置示例（目标效果）

### 回测专员
- 动作：`task:view/create/cancel/restart/delete`
- 作用域：`task_type in ['backtest_training']`
- 页面：`backtest:view`、`backtest:create`

### Google C3 专员
- 动作：`task:view/create/cancel/restart/delete`
- 作用域：`task_type in ['google_sheet']`（或 `version='c3'`）
- 页面：`google_sheet:c3`

### 管理员
- 动作：全量
- 作用域：`all`
- 页面：全量

## 9. 验收标准（必须通过）
- 回测角色看不到任何 Google 任务，且不能操作 Google 任务 ID。
- Google C3 角色看不到回测任务，且不能操作回测任务 ID。
- 手输 URL、直接调用 API 都会被后端拦截并返回缺失权限信息。
- 菜单、列表、详情、操作按钮行为一致，无“前端可见后端失败”大面积冲突。

---

本方案目标：把鉴权从“路由字符串判断”升级为“资源级策略判断”，让后续新增任务类型时不再污染已有权限。
