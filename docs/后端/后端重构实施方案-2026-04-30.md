## 1. 目标

本方案不是推倒重来，也不是切换框架。  
目标是基于当前 Flask + 任务平台架构，完成一轮**低风险、可分阶段上线**的后端整治，重点解决：

- 任务编排职责过载
- 调度器/看门狗实现债
- 运行态状态源过多
- 路由层聚合逻辑过重
- 测试体系缺失

重构原则：

1. 先收敛边界，再谈大重写
2. 先修正确性和稳定性，再做优雅
3. 每一阶段都要可单独上线
4. 不同时改任务执行逻辑和页面协议，避免联动爆炸

## 3.1 修复 `scheduler_service.py` 残留逻辑

### 问题

`app/services/scheduler_service.py` 中 `_release_task_lock()` 后面混入了不可达旧逻辑，属于明显实现债。

### 目标

- 清除不可达代码
- 明确调度器职责
- 保证行为不变

### 具体改法

1. 删除 `_release_task_lock()` 方法体内残留的旧“运行具体任务函数”逻辑
2. 保留真正有效的执行路径：
   - APScheduler 触发
   - DB 乐观抢锁
   - `subprocess.Popen`
   - `scheduled_task_worker.py` 执行
   - worker 释放锁
3. 给 `SchedulerService` 补一个模块级说明，明确：
   - 主进程只负责任务触发和抢锁
   - 子进程负责实际清理任务

### 验收标准

- 文件中不存在不可达旧分支
- “立即执行”和 cron 触发都还能正常运行
- 锁能正常释放

## 3.2 修复看门狗配置未生效

### 问题

`app/services/task_watchdog.py` 读取了 `watchdog_interval_seconds`，但最终实际睡眠仍是固定 20 秒。

### 目标

- 让配置真正生效
- 避免误导运维和后台配置页

### 具体改法

1. 用 `interval_seconds` 替换硬编码 `20`
2. 保留最小保护值，比如 `max(interval_seconds, 5)`
3. 日志里打印实际生效配置：
   - interval
   - log_timeout
   - enabled

### 验收标准

- 配置修改后无需改代码即可改变 watchdog 周期
- 日志能看到 watchdog 使用的真实配置

## 3.3 给认证降级路径加 fail-fast

### 问题

当前存在两个危险默认行为：

- `JWT_SECRET_KEY` 缺失时用默认值
- `AUTH_ENABLED=false` 时注入全权限 mock 用户

### 目标

- 开发环境保留便利
- 非开发环境默认安全

### 具体改法

1. 在 `create_app()` 或 `bootstrap_app()` 阶段增加安全检查
2. 规则建议：
   - `APP_ENV != development` 时，禁止默认 JWT secret
   - `APP_ENV != development` 时，禁止 `AUTH_ENABLED=false`
3. 启动失败时直接报错，不要 silent fallback

### 验收标准

- 开发环境不受影响
- 测试/生产环境配置错误时应用拒绝启动

## 3.4 明确任务收尾时的状态优先级

### 问题

当前任务状态涉及：

- DB `Task.status`
- `running_tasks`
- `task_stop_events`

已经有收尾逻辑，但还不够显式。

### 目标

把“取消优先于完成”“DB 状态优先于线程缓存”写成统一规则。

### 具体改法

1. 在 `TaskManager` 中整理统一状态收尾函数
2. 固定规则：
   - DB 已是 `cancelled`，不得再回写 `completed`
   - stop_event 已触发，执行结果必须走取消分支
   - 线程退出时必须清理内存态
3. 给 `restart_task()` / `cancel_task()` / `_finalize_task_execution()` 增加统一注释

### 验收标准

- 不再出现明显的 cancelled/completed 竞争回写
- 任务线程退出后内存态始终释放

## 4. P1 结构收敛

执行状态：已完成

## 4.1 拆分 `TaskManager`

### 目标

保留 `TaskManager` 门面，但把大块职责拆出去。

### 建议拆分

1. `app/services/task_creation_service.py`
   - `create_task`
   - config 规范化
   - token 配置预处理

2. `app/services/task_runtime_service.py`
   - `start_task`
   - 执行器线程启动
   - 运行态句柄管理

3. `app/services/task_restart_service.py`
   - `cancel_task`
   - `restart_task`
   - 状态恢复与重启判定

4. `app/services/task_query_service.py`
   - `get_task_status`
   - `get_all_tasks`
   - `get_tasks_paginated`
   - `check_local_task_status`

5. `app/services/task_batch_service.py`
   - `batch_create_and_start_task`
   - C31 参数组合与子任务透传

### `TaskManager` 最终角色

- 只做 facade
- 保持对旧路由接口兼容
- 内部委托给新 service

### 推荐步骤

1. 先抽 query service
2. 再抽 batch service
3. 再抽 runtime/restart
4. 最后让 `TaskManager` 退化为门面

### 验收标准

- `task_manager.py` 行数降到 500 行以内
- 现有 API 协议不变
- 核心逻辑可单独测试

## 4.2 增加任务配置 DTO / Schema

### 问题

当前任务配置大量以 `dict[str, Any]` 贯穿：

- 路由层
- `TaskManager`
- service 执行层

这会让字段演进成本很高，也不利于 C31/C3 对齐。

### 目标

先不强行全量 dataclass 化，但至少为关键任务类型建立 schema。

### 建议方案

新增目录：

- `app/schemas/`

建议首批对象：

- `google_sheet_task_config.py`
- `google_sheet_c31_batch_config.py`
- `backtest_training_task_config.py`

### 第一阶段只做什么

1. 负责字段读取与默认值归一化
2. 提供 `from_dict()` / `to_dict()`
3. 把 `market_type`、`end_date`、`token_id`、`spreadsheet_id` 这类关键字段集中定义

### 验收标准

- 关键字段不再散落各处手写 `get()`
- C31 到 C3 的透传字段有单一来源定义

## 4.3 抽离管理后台聚合查询

### 问题

`app/routes/admin.py` 当前直接承担：

- 任务运行态详情拼装
- 结果汇总
- 最近日志
- dashboard 统计

### 目标

让路由层只负责：

- 接收请求
- 权限校验
- 返回 JSON / 页面

### 建议拆分

新增：

- `app/services/admin_dashboard_query_service.py`
- `app/services/task_runtime_view_service.py`

职责划分：

- `admin_dashboard_query_service`
  - dashboard overview
  - 管理台统计
- `task_runtime_view_service`
  - stop confirmation
  - result summary
  - runtime detail DTO

### 验收标准

- `admin.py` 行数明显下降
- 查询拼装逻辑可复用
- 后续优化 SQL 时不必改路由

### 当前进度

- 已完成 `TaskRuntimeViewService` 落地
- 已把 dashboard overview 和 runtime detail 从 `admin.py` 路由层移出
- 已把 stop confirmation 组装逻辑沉到运行态视图服务
- 当前事项已完成，dashboard 汇总统计已推进到聚合 SQL

## 4.4 统一启动职责

### 问题

现在启动逻辑虽已进入 `startup.py`，但仍混合了：

- schema patch
- default config init
- RBAC init
- occupancy reset
- scheduler/watchdog start

### 目标

把启动职责拆成“准备阶段”和“运行阶段”。

### 建议拆分

在 `app/startup.py` 基础上进一步整理：

1. `prepare_database_state(app)`
   - `db.create_all()`
   - schema compatibility patch
   - occupancy reset

2. `prepare_system_metadata(app)`
   - `init_config()`
   - `init_rbac()`

3. `start_background_components(app)`
   - `check_and_cleanup_dead_tasks()`
   - `init_scheduler()`
   - `init_task_watchdog()`

### 验收标准

- 启动代码可读性提升
- 每段职责清晰
- 更方便后续做 CLI 分离

## 4.5 整理 schema patch 的退出路线

### 目标

不是立刻删掉兼容补丁，而是把它们纳入生命周期管理。

### 具体改法

1. 在每个 `ensure_*_schema()` 上方注明：
   - 对应补丁目的
   - 对应迁移版本
   - 计划移除条件
2. 在文档中建立表格：
   - 补丁函数
   - 对应正式迁移
   - 是否已可废弃

### 验收标准

- 团队知道哪些是长期结构，哪些只是过渡兼容

## 7. 每阶段风险提示

## P0 风险

- auth fail-fast 可能影响已有开发流程
- watchdog 周期调整可能暴露隐藏竞态

控制方式：

- 先在 development 之外启用严格检查
- watchdog 改动先保守配置

## P1 风险

- 拆 service 时容易把隐式共享逻辑拆散
- C31 子任务透传字段最容易回归

控制方式：

- 先补 batch 透传测试
- 保留旧接口，内部渐进委托

## P2 风险

- 清理历史测试文件时可能误删仍有人手工依赖的脚本

控制方式：

- 先迁移到 `tests/manual/`，不要直接删除

## 9. 最终建议

这套系统当前最需要的不是换成 FastAPI，也不是立刻上 Celery，而是：

- 把任务编排边界收清
- 把后台恢复逻辑收清
- 把测试和安全底座补上

换句话说：

**先让现有系统“更稳、更清晰、更可改”，再决定要不要升级技术栈。**
