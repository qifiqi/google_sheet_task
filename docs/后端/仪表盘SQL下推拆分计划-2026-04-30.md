# 仪表盘 SQL 下推拆分计划

更新时间：2026-04-30

执行状态：已完成

关联代码：

- `app/services/task/runtime_view.py`
- `app/routes/admin.py`
- `app/utils/task_authorization.py`

关联背景：

- 当前 `TaskRuntimeViewService.build_dashboard_overview()` 已完成第一轮优化：
  - 先计算当前用户可访问的 `task_type`
  - 再把 `task_type` 过滤条件下推到 SQL
- 但统计汇总仍然存在“把允许类型下的全部任务加载到内存，再做 Counter 和趋势计算”的问题

---

## 1. 目标

把 dashboard overview 从“查询结果集后在 Python 聚合”继续演进为“由数据库完成大部分聚合”，降低以下风险：

- 任务量变大后后台首页变慢
- 单次请求占用过多内存
- 聚合逻辑分散，后续难以继续扩展

本次计划不改变接口协议，不改变前端字段结构，只优化后端查询实现。

---

## 2. 当前问题拆解

当前 `build_dashboard_overview()` 仍然做了这些内存聚合动作：

1. 查询当前允许 `task_type` 下的全部任务
2. 在 Python 中统计：
   - `summary`
   - `status_distribution`
   - `task_type_distribution`
   - `daily_trend`
3. 再额外查询：
   - 最近 10 个任务
   - 最近 6 个运行中任务

其中真正需要全量对象的，其实只有：

- `recent_tasks`
- `active_tasks`

其余都更适合在 SQL 层完成。

---

## 3. 拆分原则

### 3.1 先聚合，后对象化

凡是以下信息，都优先用聚合 SQL：

- `count(*)`
- `group by status`
- `group by task_type`
- 按日期桶聚合的创建/完成趋势

### 3.2 保持路由零感知

SQL 下推全部收口在：

- `TaskRuntimeViewService`

必要时再进一步拆到：

- `TaskDashboardQueryService`

`admin.py` 不直接感知任何 SQL 细节。

### 3.3 保持权限过滤前置

无论采用哪种聚合 SQL，第一步都必须先得到：

- `allowed_task_types`

所有 SQL 聚合都以：

- `Task.task_type.in_(allowed_task_types)`

为前置条件，不能在聚合后再回 Python 过滤。

## 4. 分阶段实施方案

## Phase 1

执行状态：已完成

目标：把 summary 和 distribution 下推到 SQL。

### 需要改造的统计项

1. `summary.total_tasks`
2. `summary.completed_tasks`
3. `summary.running_tasks`
4. `summary.error_tasks`
5. `summary.cancelled_tasks`
6. `summary.pending_tasks`
7. `status_distribution`
8. `task_type_distribution`

### 推荐实现

#### 方案 A：分两次聚合查询

1. `status` 聚合：

- `select Task.status, count(*)`
- `group by Task.status`

2. `task_type` 聚合：

- `select Task.task_type, count(*)`
- `group by Task.task_type`

优点：

- 可读性高
- SQLAlchemy ORM 写法简单
- 对当前项目最稳

### Phase 1 验收标准

- `build_dashboard_overview()` 不再为 summary/distribution 加载全部任务对象
- summary 结果和现有接口字段完全一致
- 权限过滤逻辑不变

---

## Phase 2

执行状态：已完成

目标：把最近 7 天趋势下推到 SQL。

### 需要改造的统计项

1. `daily_trend[].created`
2. `daily_trend[].completed`

### 推荐实现

分成两条聚合 SQL：

1. 创建趋势
   - 基于 `Task.created_at`
   - 限定最近 7 天
   - `group by date(created_at)`

2. 完成趋势
   - 基于 `Task.end_time`
   - 限定最近 7 天
   - `status == completed`
   - `group by date(end_time)`

查询结果返回后，再在 Python 层补齐缺失日期，生成稳定的 7 天窗口。

### 注意事项

- 由于项目当前数据库可能是 SQLite，也可能是其他方言，日期截断写法要保持兼容
- 优先采用 SQLAlchemy `func.date(...)`
- 不要为了“更漂亮”的 SQL 重构引入方言分支

### Phase 2 验收标准

- `daily_trend` 仅用聚合结果 + 空桶补齐生成
- 不再依赖任务全集逐条遍历

---

## Phase 3

执行状态：已完成

目标：把 dashboard 逻辑正式从运行态视图服务中分离。

### 建议新增

- `app/services/task/dashboard_query.py`

建议职责：

1. `get_allowed_task_types(user)`
2. `get_status_distribution(allowed_task_types)`
3. `get_task_type_distribution(allowed_task_types)`
4. `get_daily_trend(allowed_task_types, days=7)`
5. `get_recent_task_models(allowed_task_types, limit=10)`
6. `get_active_task_models(allowed_task_types, limit=6)`

然后由 `TaskRuntimeViewService` 只负责：

- `serialize_task_runtime()`
- `build_stop_confirmation()`
- `build_result_summary()`

### Phase 3 验收标准

- dashboard 聚合查询与任务运行态详情彻底解耦
- `runtime_view.py` 聚焦运行态 DTO
- dashboard 查询后续优化不再碰运行态细节

## 5. 建议 SQL/ORM 拆分顺序

建议按最小风险顺序推进：

1. 先抽 `status_distribution`
2. 再抽 `task_type_distribution`
3. 再抽 `summary`
4. 最后抽 `daily_trend`

原因：

- 前三项最容易验证结果一致
- `daily_trend` 涉及日期桶和空桶补齐，最容易引入边界问题

## 7. 风险提示

### 风险 1

如果直接一次性把所有统计都改成复杂 SQL，排查错误会很困难。

控制方式：

- 严格按 Phase 1 -> Phase 2 -> Phase 3 分步推进

### 风险 2

不同数据库方言的日期聚合可能表现不同。

控制方式：

- 优先保持 SQLite 兼容
- 统一用 SQLAlchemy 表达式，不写裸 SQL

### 风险 3

为了追求 SQL 下推，把权限过滤放回 Python，会造成安全回退。

控制方式：

- 所有聚合查询都必须以 `allowed_task_types` 为前置过滤条件
